A command like this:

    cp *.txt dir

invokes an executable. It is implemented using the bash operator, with
arguments [cp, *.txt, dir].

I've gone back and forth on quoting args, as quoting changes
semantics, only sometimes in desirable ways. E.g., quoting prevents a
filename with embedded whitespace from being interpreted as two
separate tokens. But it also prevents globs from being interpreted.

It seems best to pass on the command line exactly as is. Parsing
removes quotes, and then adding quotes tries to restore user meaning,
which is hopeless.


BUT: Sometimes we want interpretation of args passed to bash.

Example: Expression evaluation. EXT is a variable bound to a string,
(e.g. it stores a file extension):

  cp *(EXT) dir



In bash:

    jao@cheese:~/git/marcel/test$ H=hello
    jao@cheese:~/git/marcel/test$ echo $H
    hello
    jao@cheese:~/git/marcel/test$ echo "$H"
    hello
    jao@cheese:~/git/marcel/test$ echo '$H'
    $H

So $H and "$H" evaluated the $H expression, but '$H' does not.


Simple marcel rule that gets both preservation of quotes and
expression evaluation:

- Keep command line as is, quotes and everything.

- But evaluate Expressions, i.e., (...).

bash had (or has?) lots of bugs. Not really sure, since some bugs
don't show up in unit tests, which run directly (not via jobs). Also,
the ProcessOutputHandler mechanism is clunky and dumb -- a thread to
consume each of stdout and stderr.

Redesign: Use selectors module to get input from stdout and stderr.

Requirements:

- Non-interactive:

  - Works with no stdin

  - Works with stdin

  - stdout and stderr content delivered.

- Interactive

  - Operates by side effect. 

  - process runs

  - marcel blocks until completion.


Running a pipeline:

- setup

- run: first op: receive

- flush

- cleanup

If bash is one of the ops, we don't know how the bash command behaves
wrt input and output. 

- It could be read one input, write out output, repeating.

- It could accumulate all input and then write all output (e.g. sort).

- Or any other pattern.

Bash input comes from marcel pipeline. Bash output is received by
selector output and sent downstream. Bash execution CANNOT BLOCK
waiting for execution to complete, because that would block the
pipeline leading into it, and input could not accumulate (as required
for the sort case).

*** So selector loop must be in a thread.

*** BUT: if bash process doesn't block, then we cleanup and have a
    race with execution.
